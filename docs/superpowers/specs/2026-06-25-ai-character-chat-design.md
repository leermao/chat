# AI 角色聊天应用设计文档

## 目标

实现一个可用于面试验收的真实 AI 角色聊天应用。用户可以浏览 200 个以上 AI 角色，角色列表支持分页加载和虚拟化渲染；用户进入某个角色后可以进行真实 AI 对话，AI 回复以逐字流式方式显示；每个角色的聊天记录独立持久化保存，刷新、切换角色后仍能恢复。

项目需要尽量复刻原型录屏的视觉与交互：浅色背景、淡紫色强调、左侧导航、角色卡片网格、聊天页顶部栏、聊天历史列表。

## 技术栈

- 前端：React + Vite + TypeScript
- 后端：Node.js + Express + TypeScript
- 数据库：SQLite
- AI 接口：OpenRouter，使用 OpenAI-compatible Chat Completions stream API

采用单仓库前后端分离结构：

```text
client/  前端应用
server/  后端 API、AI 调用、SQLite 数据持久化
```

## 配置要求

API Key 只能通过后端 `.env` 读取，不能写死在代码里，也不能提交到仓库。

后端 `.env` 需要包含：

```text
OPENROUTER_API_KEY=sk-or-v1-xxxxx
AI_BASE_URL=https://openrouter.ai/api/v1
AI_MODEL=deepseek/deepseek-v4-flash
PORT=3001
```

仓库只提交 `.env.example`，不提交真实 `.env`。

## 页面与路由

### 角色广场 `/`

桌面端布局尽量复刻原型：

- 左侧固定侧栏，包含产品名、角色广场入口、聊天历史入口、底部快捷历史按钮。
- 主区域顶部显示标题“角色广场”和已加载数量。
- 顶部有介绍区，文案接近“发现你的专属对话伙伴”。
- 角色卡片三列网格展示。
- 每张卡片包含头像、角色名、简介、编号标签和“开始聊天”入口。
- 滚动到底部附近时自动请求下一页。
- 列表使用虚拟化渲染，避免一次性渲染 200+ 个角色 DOM。

H5 布局：

- 角色卡片改为单列。
- 左侧侧栏隐藏。
- 顶部显示页面标题。
- 底部提供“角色广场 / 聊天历史”的导航入口。

### 聊天页 `/chat/:characterId`

桌面端布局：

- 左侧仍保留侧栏。
- 主区域顶部栏包含返回、角色名、角色简介、历史、清空。
- 正文顶部展示角色信息条。
- 消息区展示当前角色独立聊天记录。
- 用户消息靠右，AI 消息靠左。
- 底部输入框固定，提示支持 `**粗体**` 和 `*斜体*`。
- 发送按钮在输入为空或 AI 回复中时禁用。

行为：

- 首次进入某个角色聊天页时，后端确保开场白存在。
- 开场白作为 assistant 消息保存到聊天记录中。
- 刷新页面后开场白仍保留。
- 用户发送消息后，前端立即展示用户消息。
- 后端调用 OpenRouter，流式转发 AI 回复。
- 前端接收后按字符节奏逐字显示，确保视觉上不是整段突然出现。
- 流式结束后，后端保存完整 AI 回复。
- 如果 AI 调用失败，前端展示错误状态，用户可以继续输入重试。

H5 布局：

- 顶部保留返回按钮、角色名、历史入口、清空入口。
- 消息区和输入框适配小屏，不出现横向滚动。
- 底部导航允许回到角色广场和聊天历史。

### 聊天历史 `/history`

桌面端布局：

- 左侧保留侧栏。
- 主区域顶部显示“聊天历史”和副标题。
- 提供“发现新角色”按钮返回角色广场。
- 两列展示已经聊过的角色。
- 每条历史包含头像、角色名、最近一条消息摘要、更新时间。
- 点击历史项进入对应聊天页。

H5 布局：

- 历史列表改为单列。
- 顶部保留返回或发现新角色入口。

## 后端接口

### `GET /api/characters?page=1&pageSize=24`

分页返回角色列表。

响应字段：

- `items`：角色数组
- `page`：当前页
- `pageSize`：每页数量
- `total`：总角色数
- `hasMore`：是否还有下一页

### `GET /api/characters/:id`

返回单个角色详情。

### `POST /api/conversations/:characterId/ensure-greeting`

确保指定角色存在开场白。

- 如果该角色没有任何消息，则创建一条 assistant 开场白。
- 如果已经有消息，则不重复创建。
- 返回当前角色消息列表。

### `GET /api/conversations/:characterId/messages`

返回指定角色的聊天记录，按创建时间升序排列。

### `POST /api/conversations/:characterId/stream`

发送用户消息并返回 AI 流式回复。

请求体：

```json
{
  "content": "用户输入内容"
}
```

处理流程：

1. 校验角色存在。
2. 保存用户消息。
3. 读取该角色最近若干条消息作为上下文。
4. 根据角色名和简介构造 system prompt。
5. 调用 OpenRouter stream。
6. 将 AI token/chunk 以 `text/event-stream` 转发给前端。
7. 累积完整 assistant 内容。
8. 流结束后保存完整 assistant 消息。

如果 OpenRouter 返回错误，后端向前端发送错误事件，不保存空的 assistant 消息。

### `DELETE /api/conversations/:characterId`

清空指定角色的全部聊天记录。

清空后再次进入该角色聊天页时，`ensure-greeting` 会重新创建开场白。

### `GET /api/conversations`

返回已经聊过的角色列表。

每项包含：

- 角色信息
- 最近一条消息摘要
- 最近更新时间

## 数据模型

### `characters`

- `id`：主键
- `name`：角色名
- `description`：简介
- `avatarColor`：头像颜色
- `avatarUrl`：头像地址，可为空
- `greeting`：开场白
- `createdAt`：创建时间

后端启动时检查角色数量，如果少于 220 个，则自动 seed 示例角色。角色内容可以按编号生成，但名称、简介、头像颜色要有一定变化，方便验证列表效果。

### `messages`

- `id`：主键
- `characterId`：角色 ID
- `role`：`user` 或 `assistant`
- `content`：消息内容
- `createdAt`：创建时间

不同角色通过 `characterId` 隔离上下文。

## AI 上下文策略

每次请求只使用当前角色的消息作为上下文，避免不同角色聊天记录互相影响。

上下文包含：

1. system prompt：说明当前角色身份、简介、回复风格。
2. 当前角色保存后的最近 20 条消息，其中包含最新用户消息。

如果历史消息很多，只取最近 20 条，避免 prompt 过长影响响应速度。

## Markdown 渲染

前端消息内容使用 Markdown 渲染组件处理，至少支持：

- `**粗体**`
- `*斜体*`

Markdown 渲染需要避免直接插入不可信 HTML，默认不启用原始 HTML。

## 流式显示策略

OpenRouter 返回的 chunk 大小不可控。为了满足“一个字一个字地流式显示”，前端需要做显示缓冲：

- 网络层持续接收后端流。
- 将收到的文本追加到待显示队列。
- UI 层使用定时器按字符追加到当前 assistant 消息。
- 流结束且队列清空后，消息状态变为完成。

这样即使后端一次返回多个字符，用户看到的仍是逐字输出。

## 错误处理

- 缺少 `OPENROUTER_API_KEY`：后端启动或 AI 请求时返回明确错误。
- OpenRouter 请求失败：聊天页展示错误提示，不清空用户已输入消息。
- 网络中断：停止当前流式状态，允许用户重新发送。
- 角色不存在：返回 404，前端显示可返回角色广场的错误页。
- 清空记录失败：保留当前消息列表并提示失败。

## 验收点

- 首页至少有 200 个角色。
- DevTools 能看到分页请求，而不是一次性请求所有角色。
- 列表使用虚拟化，页面中不会同时存在 200+ 个角色卡片 DOM。
- 点击角色进入聊天页。
- 首次进入新角色时出现开场白，并持久化保存。
- 用户能发送消息。
- AI 回复来自真实 OpenRouter 调用。
- AI 回复是流式显示，并且视觉上逐字出现。
- 用户消息和 AI 回复支持 Markdown 粗体、斜体。
- 刷新页面后聊天记录恢复。
- 不同角色聊天记录互不影响。
- 聊天历史页展示聊过的角色和最近消息。
- 点击历史项能回到对应聊天页。
- 清空某个角色后，再进入该角色会重新显示开场白。
- API Key 不出现在前端代码、构建产物或仓库文件中。
- H5 下首页、聊天页、历史页均可正常操作。

## 非目标

- 不做用户登录。
- 不做多用户数据隔离。
- 不做角色搜索和收藏。
- 不做图片上传。
- 不做部署流水线。

这些能力不属于本次面试题必要范围，避免影响核心功能完成质量。
